Grails3 Restful 入门

前言

  Grails是运行在JVM之上，成熟稳定的JEE开发环境。与时下流行的Python、NodeJs相比在开发效率不输的情况下，同时拥有性能上的优势。IDEA中的Grails插件极大弥补了缺乏开发工具这一不足。不过，对于“全栈”这个概念，选择Grails意味着你的技术栈将会显得相对臃肿。另外，Grails中文资料相对欠缺。

坑

  如何使用Grails开发restful应用这里不谈，相关知识可以参阅官方文档。这里我只讨论遇到的坑。 Grails官方文档创建的rest应用只是最基本的用法，无法适用于商业项目。我们需要对项目进行一定程度的重构。

  • 传输参数
    有时由于业务需要，我们需要对restful接口的传输参数进行加解密。实现办法是，重写controller的基类RestfulController的getObjectToBind方法，将处理后的参数打包成map进行返回。

  • 错误json信息处理
    对于restful api调用中出现的错误，Grails自行进行了处理，并以json或xml格式文本对客户端进行了响应。这也显然不能满足项目需求，商业项目中有时会对几个端的错误定制某种约定，设计相应的错误码来标识特定的错误，特别是业务逻辑错误。所以对返回的错误信息需要改造。官方给出方案是自定义grails.rest.render.errors.VndErrorJsonRenderer的实现，并在配置文件resources.groovy中配置即可。即:
    If you wish to change the format to Vnd.Error then simply register grails.rest.render.errors.VndErrorJsonRenderer bean in grails-app/conf/spring/resources.groovy:

  不幸的是这么做是无效的。通过跟踪源码，发现其实Grails处理错误信息是通过views模块中的errors/_errors.gson模板实现的。可通过修改此文件来定制自己的错误信息json输出格式。默认的错误message是在“i18n”模块中相应的messages.properties文件中配置的。

结束

  自此，Grails-Restful-Api可以适应大部分的restful开发了。Grails相对于时下最火的NodeJs有着无法比拟的性能优势，但对于对性能要求极高的模块和项目她也是无能为力的，虽然可以用Java语言混编，但可以预见，效果不会太显著。建议通过使用Java+Rersey2进行原生开发，摒弃一切开源框架，完全轻量级。另外，开发restful项目，需要对restful有一定程度的了解。ok，祝大家愉快。
